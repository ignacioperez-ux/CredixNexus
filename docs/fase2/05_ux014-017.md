# Fase 2 · UX-014/015/016/017 — Deuda de design system y RBAC (menor)

## UX-014 — FeedbackWidget confía en la prop `canFeedback` · VERIFICADO (sin cambio)
El candado real **ya está en el servidor**: `submitKbFeedback` valida `knowledge.feedback`
(`lib/knowledge/actions.ts:63`) antes de escribir. La prop del widget solo controla **visibilidad**;
la acción está gateada (defense-in-depth). No requiere cambio de código. *(HECHO)*

## UX-015 — Alias de token solo en Claro → rompía Nexus · CORREGIDO
Los alias `--primary/--primary-hover/--on-primary/--surface/--surface-alt/--ink/--ink-strong`
existían **solo** en el tema Claro; usarlos sin fallback rompía Nexus. Se añaden a `:root`
referenciando los tokens semánticos (`--primary: var(--accent)`, `--surface: var(--card)`, …), por lo
que resuelven correctamente en **ambos** temas. Claro conserva sus valores literales (mayor
especificidad por orden). `app/globals.css`.

## UX-016 — Hex sueltos fuera de tokens · CORREGIDO
- `help-fab.tsx`: sombra roja `rgba(227,6,19,.5)` → `var(--sh-fab)`.
- `command-menu.tsx`: overlay `rgba(0,0,0,.42)` → `var(--overlay)`; sombra modal → `var(--sh-modal)`.
- Nuevos tokens en `globals.css`: `--overlay`, `--sh-modal`, `--sh-fab`.
- *(Nota: el fallback de `wordmark.tsx` usa un degradado de marca al fallar la imagen — estado
  degradado poco frecuente; se deja como está.)*

## UX-017 — `partner_admin` sin permisos · DIFERIDO (decisión de producto/RBAC)
El rol `partner_admin` ("Admin partner") tiene **cero permisos** → inerte. **No se modifica aquí**:
otorgar permisos es una decisión de RBAC/seguridad (§2.4). No existe un permiso *scoped* de
"gestionar usuarios de mi organización" (dar `user.manage` sería cross-tenant, excesivo).
**Recomendación:** el arquitecto define qué hace `partner_admin` (p.ej. baseline `partner_user` +
un permiso nuevo acotado de gestión de usuarios de su party) y se implementa con migración + RLS.

**Verificación:** `npm run build` ✅ · `vitest` **250/250** ✅.
