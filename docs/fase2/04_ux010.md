# Fase 2 · UX-010 — Categorías del catálogo como datos maestros (i18n ES/EN)

**Problema:** `service_item.category` era **texto libre en español**, mostrado en crudo (en EN se
veía en español). Sin maestro, sin i18n, sin CRUD (viola §10/§11).

**Solución (§10 completo, decisión del arquitecto): maestro `service_category`.**

### Base de datos (`sql/0087_service_category.sql`, aplicada)
- Tabla `service_category`: `code`, `name_es`, `name_en`, `sort_order`, `status` (soft-delete),
  `tenant_id`, auditoría (`created_by/updated_by/at`). Constraints: `UNIQUE(tenant_id, code)`,
  CHECK status, NOT NULL. **RLS** por tenant + triggers de auditoría y `updated_at`.
- FK `service_item.category_id` → `service_category` (ON DELETE SET NULL) + índice.
- Seed con i18n (acceso/datos/general) para **todo tenant con ítems**; **backfill** por `code`
  (0 ítems huérfanos verificado).

### Aplicación
- `lib/catalog/queries.ts`: `CatalogItem` trae `category_code/name_es/name_en` (join); nuevo
  `listServiceCategories`.
- `lib/catalog/actions.ts`: `createItem` usa `categoryId` (valida referencia); nuevos
  `createCategory` / `setCategoryStatus` (guard `service_catalog.manage`, duplicado → DUPLICATE).
- `lib/catalog/validation.ts`: `validateItem` exige `categoryId`; nuevo `validateCategory`.
- **UI:** `catalog-grid` agrupa/muestra la categoría **localizada** (fallback legacy); `item-form`
  usa **select desde el maestro**; `item-manager` muestra nombre localizado + **gestión CRUD de
  categorías** (alta + activar/desactivar) — pantalla de Datos Maestros (§10.5).
- i18n ES/EN para el admin de categorías; los nombres de categoría salen del maestro (no del
  diccionario) → sin hardcode.

### Fuera de alcance (honesto)
- Los `label` de `form_schema` son **contenido definido por el admin** por ítem (no chrome): no se
  i18n-izan (como el contenido de un artículo). Documentado.
- `incident_category` (portal) queda para una pieza análoga si se desea (usa su propia maestra).
